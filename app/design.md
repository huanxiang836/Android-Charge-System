# 安卓奶茶店收银系统 Figma 方案

## Summary
基于当前空白 Android 模板工程，本次先不做代码实现，先完成一套可直接指导开发的 Figma 原型，目标是常见 `15.6` 寸安卓横屏 POS 收银终端，风格采用标准商用 POS，范围覆盖主收银流：`点单首页`、`结算区/结算弹层`、`小票预览`。

交付结果应能直接支撑后续 Android 落地，重点保证高频点单操作少跳转、主路径清晰、收银员单手或双手连续操作顺畅。

## Key Changes
1. 新建一份 Figma 收银终端文件，按桌面横屏 `1920×1080` 建立主画板，并固定为收银主场景。
2. 设计主收银首页，布局按“三栏高频操作”组织：
    - 左侧：商品分类与快捷筛选，如奶茶、果茶、小料、加冰加糖、套餐。
    - 中间：商品卡片区，支持快速点选、热销标识、规格入口、搜索入口。
    - 右侧：当前订单区，展示已选商品、数量调整、单项删减、应付金额、清空与挂起预留位。
3. 设计价格计算与结算区：
    - 订单区底部固定显示商品总数、原价、小计、优惠占位、应付金额。
    - 结算动作包含现金、扫码、混合支付占位，其中本次以基础收银为准，不展开会员价和优惠券流程。
    - 结算页可做为右侧扩展态或中心弹层，但最终统一一种交互，不同时保留两套。
4. 设计小票生成相关页面：
    - 提供“小票预览”画板，内容包括门店名、订单号、时间、商品明细、数量、金额汇总、支付方式、找零/实收信息、欢迎语。
    - 同时给出“打印前预览态”和“支付完成后查看小票入口”的衔接状态。
5. 在 Figma 内同步沉淀最小组件集，便于后续 Android 映射：
    - 分类 Tab
    - 商品卡片
    - 订单行
    - 数量步进器
    - 主按钮/次按钮
    - 金额信息块
    - 支付方式按钮
    - 小票信息行
6. 组件与样式约束统一为高可读 POS 规范：
    - 大按钮、大字号、高对比度
    - 主操作区避免隐藏入口
    - 文本、颜色、间距、圆角形成基础 token
    - 中文文案直接按业务态命名，如“现金收款”“确认结算”“打印小票”

## Public Interfaces / Deliverables
Figma 需至少包含以下明确产物，供后续开发直接对照：
- `POS-首页点单`
- `POS-结算态`
- `POS-小票预览`
- `POS-支付完成态` 可作为同页状态分支或独立画板，但需明确状态切换
- `基础组件区`，放置按钮、商品卡、订单行、金额摘要等可复用组件
- `样式说明区`，标注字号层级、主色/状态色、栅格与边距规则

需要显式标注的交互/数据字段：
- 商品名、规格、单价、数量、小计
- 订单总件数、应付金额、实收金额、找零金额
- 订单号、下单时间，时间格式统一按 `yyyy-MM-dd HH:mm:ss`
- 小票生成入口、返回继续下单入口

## Test Plan
完成 Figma 后按以下场景自检：
- 商品连续点击 5 到 10 次时，订单区仍清晰可读，不出现挤压或滚动冲突。
- 商品数量增加、减少、删除后，金额汇总区域层级稳定，主支付按钮始终可见。
- 在 `1920×1080` 横屏下，收银员从选品到结算不需要跨页面往返。
- 小票预览包含完整基础字段，且金额、支付方式、时间信息有固定位置。
- 中文文案统一、清晰，无英文占位文案残留。
- 主收银路径从“选商品 → 看合计 → 结算 → 查看小票”可以在原型里完整走通。

## Assumptions
- 本次只做 Figma，不改动当前 Android 工程代码。
- 先按单门店基础收银设计，不包含会员、优惠券、退款、交班、库存、真实热敏打印机适配。
- 小票先定义为应用内预览与生成界面，后续代码阶段再决定接蓝牙或 USB 打印。
- 视觉采用标准商用 POS 路线，优先效率与可读性，不做强品牌化装饰。
- 若当前没有现成 Figma 文件，则默认新建一份独立设计文件并在其中完成上述画板与组件。
